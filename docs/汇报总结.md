# inspection-flask 阶段汇报总结

## 一、项目目标

本阶段工作的核心目标，是围绕“加油站场景下未穿工服检测”搭建一套可运行的后端检测链路，将原有工程逐步收敛到适配 YOLOv11 的实现框架中，并完成从模型推理、规则判定到告警存证的基本闭环。

结合当前代码实现，系统希望完成的业务流程是：

1. 从摄像头或调试输入获取画面。
2. 先做整帧人员检测，筛出监管对象。
3. 对每个目标做人框裁剪，再做工服检测。
4. 根据时序窗口规则判断是否存在“未穿工服”违规。
5. 对违规结果进行证据图保存与数据库落库。

---

## 二、这一阶段已完成的工作

### 1. 完成了 YOLOv11 双模型检测链路的搭建

当前代码已经形成了比较清晰的双阶段检测结构：

1. 第一阶段在 [utils/models.py](../inspection-flask/utils/models.py) 中封装人员检测模型。
2. 第二阶段在同一模块中封装工服检测模型。
3. 主链路采用“整帧检测人员 -> 裁剪人员区域 -> 检测工服”的处理方式。

这一结构已经和当前的数据集形式基本对齐。结合现有标注样例，二阶段数据集主要是对画面中正确穿戴工服的人体区域标注 `clothes`，因此当前工程已经将工服标签收敛到单类 `clothes` 配置。

### 2. 完成了主配置项的集中化管理

当前关键推理参数已经集中到 [settings.py](../inspection-flask/settings.py)，包括：

1. 权重路径与置信度阈值。
2. 输入分辨率 `IMGSZ`。
3. 监管对象标签 `MONITORED_PERSON_LABELS`。
4. 工服标签与合规模式配置。
5. ROI、时序窗口、告警抑制等参数。

这样做的意义是，后续模型更新或规则调整不需要大面积改代码，已经具备了继续迭代的基础。

### 3. 完成了 Flask 实时检测框架的主干串联

围绕 Flask 服务，当前已经实现了以下核心模块：

1. 应用初始化与模型加载，在 `applications/__init__.py`。
2. 检测线程管理，在 `applications/common/hk_custom_threading_plus.py`。
3. 取帧线程管理，在 `applications/common/hk_recorder_threading.py`。
4. 摄像头启停与违规记录接口，在 `applications/view/system/hk_camera.py`。

这说明系统已经不只是“单独跑模型”，而是具备了一个后端检测服务的基本形态。

### 4. 完成了违规规则模块与存证链路

在规则层，当前已经形成了以下结构：

1. `violation_module/vio_workwear_missing.py` 负责未穿工服规则判定。
2. `violation_module/base.py` 负责告警证据图选择、绘制和落库。
3. `applications/models/admin_violate_photo.py` 对应违规记录表结构。

这意味着工程已经打通了“检测结果 -> 规则判断 -> 违规记录”这一条关键业务闭环。

### 5. 对规则口径做了本轮重点修正

和前一版相比，本阶段后半段已经完成了几项关键逻辑优化：

1. ROI 判断从“整框完全在 ROI 内”改成“中心点在 ROI 内”。
2. 工服合规逻辑从固定写死改成配置驱动。
3. 时序判定开始从“按帧统计”过渡到“按轨迹统计”。
4. 辅助诊断工具 `main.py` 已同步补充核心配置输出，便于离线排查。

这些修正说明当前系统已经从“能跑”逐步进入“开始追求逻辑严谨”的阶段。

## 三、当前系统现状

### 1. 当前链路已经具备的能力

从现有代码看，系统已经可以完成：

1. 加载 YOLOv11 风格模型权重。
2. 对整帧画面做人目标检测。
3. 对人框区域做工服检测。
4. 在时间窗口内做违规规则判断。
5. 保存违规证据图并写入违规记录。
6. 通过 Flask 接口查询摄像头和违规记录。

### 2. 当前数据集与规则的匹配关系

结合你给出的样例图，当前数据集的特点比较明确：

1. 标注重点是“穿了工服的人”，标签形式偏向单类 `clothes`。
2. 数据并没有显式区分 `customer` 和 `worker`。
3. 因此当前代码更适合做“检测是否命中工服特征”，而不适合直接做“天然区分谁必须穿工服”。

这也是为什么目前系统虽然能做“未穿工服回推”，但在共享区域场景下仍存在误报风险。

### 3. 当前实现的阶段定位

综合来看，当前工程适合被定义为：

“已完成后端检测主链路搭建，并进入规则正确性与场景适配优化阶段。”

也就是说，这一阶段的重点成果不是“已经完全上线”，而是“核心链路已搭好，下一阶段进入精化和联调”。

## 四、当前仍存在的主要问题

### 1. 监管对象边界仍然不够清晰

尽管代码已经预留了 `MONITORED_PERSON_LABELS` 配置，但当前实际仍然是把 ROI 内的 `person` 作为监管对象。由于数据集中没有“谁是员工、谁是顾客”的监督信号，在加油站前庭这种共享区域里，customer/worker 混淆仍然存在。

### 2. 新引入的轨迹规则还没有完全闭合

本轮已经加入了基于 IoU 的简易跟踪器，但目前还有两个问题：

1. 只出现 1 帧的目标也可能直接触发违规。
2. 当前取帧周期较长，和 IoU 跟踪适用的短时连续帧场景并不完全匹配。

这说明“按轨迹判定”的方向是正确的，但这部分还处在第一版逻辑阶段。

### 3. 告警证据与触发对象可能不一致

规则层已经开始计算具体触发的轨迹，但当前存证逻辑还没有和该轨迹完全绑定，存在“真正触发的是 A，人证据图保存成 B”的风险。这个问题会直接影响后续汇报、复核和系统可信度。

### 4. 真实摄像头联调能力还需要继续补齐

目前采集链路中，`hk_recorder_threading.py` 仍主要依赖 `frame_path` 调试图方式获取输入，说明离线验证已经能做，但真实视频流和生产采集能力还没有完全补齐。这是下一阶段必须推进的工程项。

## 五、下一阶段工作计划

下一阶段建议分四个方向推进。

### 1. 优先收口规则正确性

这是最核心的一步，建议优先处理以下内容：

1. 给轨迹判定增加最少出现帧数或连续违规约束。
2. 让真正触发阈值的轨迹和最终证据图严格绑定。
3. 统一面积过滤口径，避免不同模块使用不同标准。

这一步的目标，是先把“告警逻辑是否自洽”做扎实。

### 2. 提升共享场景下的监管对象约束能力

由于当前数据集没有区分 customer/worker，下一阶段需要在业务侧和数据侧同步推进：

1. 在短期内优先采用更严格的员工专属 ROI，降低共享区域误报。
2. 评估是否补充“受工服约束对象”相关监督信息。
3. 如果后续数据允许，再进一步考虑从 `person` 过渡到更细的监管对象标签体系。

这一步的目标，是逐步解决 customer/worker 混淆问题。

### 3. 补齐真实采集链路与联调

下一阶段需要尽快把“离线图调试”推到“真实摄像头输入”：

1. 接入 RTSP 或海康 SDK 的实际取帧。
2. 完成摄像头输入、线程调度、规则判定、违规落库的联调。
3. 验证多摄像头场景下的线程稳定性和调度稳定性。

这一步的目标，是让系统具备真实运行条件。

### 4. 建立阶段性评估与验收标准

建议下一阶段同步建立评估标准，而不是只凭主观观察：

1. 统计未穿工服检测的准确率、召回率、误报率。
2. 单独分析顾客误报、边缘漏报、遮挡漏报等典型错误类型。
3. 建立小规模固定验证集，作为后续每轮迭代的回归测试依据。

这一步的目标，是让后续优化有明确量化依据。

## 六、组会可直接汇报的简版结论

可以直接概括为以下四点：

1. 这一阶段已经完成了 inspection-flask 的 YOLOv11 双模型检测主链路搭建，打通了“检测、规则、存证、落库”的后端闭环。
2. 当前系统已经能够围绕 `person -> crop -> clothes -> violation` 这一流程运行，并且开始支持更灵活的配置化规则。
3. 本轮已经修正了 ROI 与时序判定中的一部分关键问题，但共享区域中的 customer/worker 混淆、轨迹判定稳定性、证据一致性仍是主要风险。
4. 下一阶段工作的重点，不是再大幅扩功能，而是优先把规则正确性、真实采集联调和场景适配做扎实。
