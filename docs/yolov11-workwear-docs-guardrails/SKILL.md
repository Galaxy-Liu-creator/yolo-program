# YOLOv11 工服检测文档约束技能

## 总则

在本项目中生成或修改代码前，先执行文档检查，再执行编码。所有实现都必须服务于“YOLOv11 加油站工人未穿戴工服检测”这一目标场景，不能继续沿用旧警务语义作为默认业务前提。

## 编码前必读文档

开始生成代码前，至少按下面顺序检查文档：

1. `../yolov11_workwear_system_design.md`

## 强制执行规则

每次编码都必须满足以下要求：

1. 以文档为规范，以代码为落地实现，文档优先级高于旧代码习惯。
2. 所有检测语义都必须围绕 YOLOv11、gas station worker、workwear missing 展开。
3. 对旧代码中残留的 police、police uniform、警服、派出所 等历史语义，默认视为待迁移遗留，不得继续扩散到新实现中。
4. 生成的代码必须逻辑严谨、可落地、可推演，不能只停留在表面重命名或演示性伪实现。
5. 代码风格必须统一，命名要稳定，尤其是 `workwear`、`worker`、`camera`、`violation`、`rule_code`、`rule_name`、`station_id`、`dept_id`、`sub_id`。
6. 不允许加入谄媚、夸张、情绪化或无信息增量的措辞，不论是在注释、日志、说明还是回复中。
7. 不允许凭空创造文档没有支持的业务规则。
8. 当旧代码与文档冲突时，以文档定义的新目标行为为准，仅保留必要兼容层。

## 实现优先级

实现代码时，按以下优先级进行判断：

1. 先保证 YOLOv11 工服检测场景的业务语义正确。
2. 再保证 person detection、workwear detection、temporal judgment、evidence output 这条链路完整。
3. 再保证数据字段语义一致，尤其是 `station_id`、`dept_id`、`sub_id`、`rule_code`、`rule_name`。
4. 最后再处理表现层、局部格式或次要结构优化。

## 输出要求

生成代码或解释代码时，遵循以下要求：

1. 表达直接、专业、克制。
2. 说明必须可核查，不能堆砌空话。
3. 注释只解释真正不明显的逻辑，不写低价值注释。
4. 对未被文档完全定义的地方，要明确写出假设前提。
5. 不要为了形式完整而输出无落地价值的代码。

本技能允许在生成代码时暂时不关注 import、包解析、模块 wiring、环境拼装等问题。不要让这些问题阻塞核心业务逻辑设计，但必须保证主流程、接口语义、判断逻辑和数据结构本身是自洽的，后续可统一补齐集成问题。

## 交付前自检

在结束编码前，至少确认以下几点：

1. 实现是否仍然严格服务于 YOLOv11 加油站工人未穿戴工服检测。
2. 是否无意间重新引入了旧警务场景的规则、标签或命名。
3. 规则判断链路是否可以真正落地，而不是停留在概念层。
4. 字段语义是否与现有项目迁移方向一致。
